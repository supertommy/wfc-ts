# Phase 4: Performance Ratchet and Learning Guide

> Part of [Backtracking Search](./README.html) · **8 story points** (≤ 8)

## Goal

Measure the cost of opt-in backtracking, protect the existing default restart path, optimize only where the data demands it, and document the feature for users and future contributors. After this phase, the implementation is ready for normal development review and the algorithm is teachable through an interactive guide.

## Skills to Reference
- `testing-gotchas` — avoid guessed timing thresholds; use ratchets grounded in measured signal/noise.
- `game-dev-gotchas` — ratchet discipline, deterministic replay, zero-allocation hot-path cleanup.
- `writing-with-tommy` — README and guide prose in Tommy's voice.
- `make-pages-interactive` — serve and monitor the learning guide surface.

## Verification Strategy

script-based ratchet + docs/browser review. The goal is to preserve default speed and make opt-in cost explicit.

## Sprints

### Sprint 0: Verification Harness
**Goal:** Define the performance and docs checks before optimization.
**Executor:** Test-writing sub-agent with context isolation.

- [ ] Task 0.1: Create `performance-test/backtracking-ratchet.ts` that reports default restart speed, opt-in backtrack speed, success rate, and backtrack counts for fixed fixtures.

- [ ] Task 0.3: Add a learning-guide build check once the guide files exist.
- [ ] Task 0.4: Confirm the ratchet script runs and prints diagnostics before thresholds are enforced.
**Verification:** The script emits diagnostics without pretending unmeasured thresholds are known.

### Sprint 1: Ratchet default restart speed and opt-in cost
- [ ] Task 1.1: Implement `performance-test/backtracking-ratchet.ts` `(2pt)`.
  - Include these scenarios:
    - existing default 2D Summer or closest existing hard input via current harness fixtures;
    - `WFCSolver` with omitted `search`;
    - `WFCSolver` with `search: { strategy: 'backtrack' }` on a hard 2D fixture;
    - `WFCSolver3D` rich pipe `6^3` and `8^3`.
  - Print JSON lines with `scenario`, `ok`, `elapsedMs`, `attempts`, `backtracks`, `cells`, and `seed`.
- [ ] Task 1.2: Establish ratchet thresholds only after measuring 20 repeated runs `(1pt)`.
  - DO NOT guess a millisecond threshold before measuring noise.
  - Use `testing-gotchas` guidance: thresholds must be separated from noise by at least 2×.
- [ ] Task 1.3: Add a package script if useful `(1pt)`:

```json
"ratchet:backtracking": "bun run performance-test/backtracking-ratchet.ts"
```

**Demo:** `bun run performance-test/backtracking-ratchet.ts` prints all scenarios and exits 0.

### Sprint 2: Optimize measured bottlenecks only
- [ ] Task 2.1: If full snapshots exceed the accepted opt-in budget, preallocate checkpoint buffers up to `maxDepth` instead of allocating arrays on every checkpoint `(2pt)`.
  - DO NOT replace snapshot restore with a partial undo trail unless a measured bottleneck remains after preallocation.
- [ ] Task 2.2: If heap rebuild dominates restore time, add a local `rebuildSelectionStructure()` helper shared by `clear()` and checkpoint restore to remove duplicate loops `(1pt)`.
  - This is a cleanup only if it reduces duplicated code and preserves behavior.
- [ ] Task 2.3: Re-run `bun run performance-test/backtracking-ratchet.ts` and record before/after numbers in the commit message `(1pt)`.
**Demo:** Ratchet script passes and default restart path stays within threshold.
