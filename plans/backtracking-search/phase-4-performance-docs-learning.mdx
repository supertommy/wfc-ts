# Phase 4: Performance Ratchet and Learning Guide

> Part of [Backtracking Search](./README.html) · **8 story points** (≤ 8)

## Goal

Measure the cost of opt-in backtracking, protect the existing default restart path, optimize only where the data demands it, and document the feature for users and future contributors. After this phase, the implementation is ready for normal development review and the algorithm is teachable through an interactive guide.

## Skills to Reference
- `testing-gotchas` — avoid guessed timing thresholds; use ratchets grounded in measured signal/noise.
- `game-dev-gotchas` — ratchet discipline, deterministic replay, zero-allocation hot-path cleanup.
- `writing-with-tommy` — README and guide prose in Tommy's voice.
- `make-pages-interactive` — serve and monitor the learning guide surface.

### Sprint 2: Optimize measured bottlenecks only — **SHIPPED / NO CODE CHANGE**
> **Decision:** No optimization was warranted. The Sprint 1 ratchet stayed below all accepted budgets, and the fixed samples showed almost no snapshot restore pressure: 2D backtrack averaged `0.05` restores with max `1`; rich 3D 6³/8³ averaged `0` restores with max `0`. Preallocating checkpoint buffers or refactoring heap rebuilds now would be speculative.

- [x] Task 2.1: If full snapshots exceed the accepted opt-in budget, preallocate checkpoint buffers up to `maxDepth` instead of allocating arrays on every checkpoint `(2pt)`.
  - DO NOT replace snapshot restore with a partial undo trail unless a measured bottleneck remains after preallocation.
  - Not triggered by measurements.
- [x] Task 2.2: If heap rebuild dominates restore time, add a local `rebuildSelectionStructure()` helper shared by `clear()` and checkpoint restore to remove duplicate loops `(1pt)`.
  - This is a cleanup only if it reduces duplicated code and preserves behavior.
  - Not triggered by measurements.
